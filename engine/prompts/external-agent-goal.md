套用方式：把任務類型指引轉成範圍、限制、風險與驗收覆蓋，不要展開成步驟。任何必要證據都必須保留在輸出內容，不可只留在內部分析。

## 最終輸出契約：goal.md

完成內部分析後，直接輸出一份可存成 `goal.md` 的 Markdown；不要加前言、分析過程、code fence 或 plan JSON。

必須使用下列結構：

# Goal
## 目標
## 背景與現況
## 範圍
## 非目標
## 限制與相容性
## 完成定義（DoD）
## 人工驗收
## 待確認事項

輸出規則：

- 上述八個標題必須全部出現、依此順序、名稱一字不差，不得省略，也不得新增其他同層級標題；沒有內容的段落寫「無」並加一句原因，原因必須指向可核對的依據——需求原文的界定，或已檢查的 repo 證據與檢查範圍（例：「無：需求僅涉及 CLI，已確認 src/ 下無對外 API 模組」）；僅寫「無特殊」「未提及」不算原因，不得留空或塞湊數內容。「目標」與「範圍」不得寫「無」——原始需求非空時必然存在目標與至少一個範圍項目，寫不出來代表分析不完整，必須先補齊再輸出；唯一例外是需求內容無法辨識為任何可分析的工作，此時「目標」寫明無法解析並引需求原文為證，「範圍」列出唯一項目「澄清需求」，並在「待確認事項」逐項列出所需資訊，不得虛構工作項目。
- Goal 要保留需求意圖與邊界，具體到能判斷是否完成，但不要提前寫成逐步任務清單。
- 「背景與現況」區分已由 repo 證實的現況、需求提供的事實與推論。
- 「範圍」每一項都要標上穩定 ID（SC-1、SC-2…）並寫明來源 AC／INV ID（直接源自需求原文者寫「需求原文」）；「完成定義」與「人工驗收」每一項都必須標明它驗證的 SC／AC ID，且範圍中每個 ID 至少被一個完成定義或人工驗收引用。已列出的每個 AC／INV ID 也必須至少出現在一個範圍、完成定義、人工驗收或非目標項目；列入非目標必須附排除依據，依附於未決事項的 ID 仍須落在上述其中一處並標注對應的待確認事項。任一邊追不到就代表輸出不完整，必須先補齊再輸出。
- 「完成定義」分開列出可由命令／exit code 判斷的機器 DoD；不可自動驗證的項目放「人工驗收」，且每一項必須附一句為何無法由命令／exit code 判斷的原因，可自動驗證的項目不得移入人工驗收。分析／文件型工作的機器 DoD 也必須寫成可執行的核對程序與通過判準（例：以 grep 或腳本逐 INV ID 檢索產物，任一 ID 缺失即失敗），無法命令化的核對明確移列「人工驗收」；不得以「產物已涵蓋」「已附行號」等靜態描述作為機器 DoD，也不要硬塞無關的 build 命令。
- 「待確認事項」逐項寫出未知資訊、影響與需要誰／何種證據決定；不得把推測寫成既定事實。沒有待確認事項時寫「無」，不要停下來反問使用者。
- 除非原需求指定其他語言，使用繁體中文。
