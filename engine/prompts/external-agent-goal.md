套用方式：把任務類型指引轉成範圍、限制、風險與驗收覆蓋，不要展開成步驟。任何必要證據都必須保留在輸出內容，不可只留在內部分析。

## 最終輸出契約：goal.md

完成內部分析後，直接輸出一份可存成 `goal.md` 的 Markdown；不要加前言、分析過程、code fence 或 plan JSON。

必須使用下列結構：

# Goal
## 目標
## 背景與現況
## 範圍
## 非目標
## 限制與相容性
## 完成定義（DoD）
## 人工驗收
## 待確認事項

輸出規則：

- 上述八個標題必須全部出現、依此順序、名稱一字不差，不得省略，也不得新增其他同層級標題；沒有內容的段落寫「無」並加一句原因，原因必須指向可核對的依據——需求原文的界定，或限定取證範圍內已讀到的 repo 證據（例：「無：需求原文明確只調整 CLI，不改對外 API」）；不得為證明「無」擴大成全 repo 負面搜尋。僅寫「無特殊」「未提及」不算原因，不得留空或塞湊數內容。「目標」與「範圍」不得寫「無」——原始需求非空時必然存在目標與至少一個範圍項目，寫不出來代表分析不完整，必須先補齊再輸出；唯一例外是需求內容無法辨識為任何可分析的工作，此時「目標」寫明無法解析並引需求原文為證，「範圍」列出唯一項目「澄清需求」，並在「待確認事項」逐項列出所需資訊，不得虛構工作項目。
- Goal 要保留需求意圖與邊界，具體到能判斷是否完成，但不要提前寫成逐步任務清單。
- 「背景與現況」區分已由 repo 證實的現況、需求提供的事實與推論。
- 只有範圍、inventory 或驗收條件需要跨多個段落／task 追蹤時，才使用 SC／AC／INV 等簡短穩定 ID，並讓每個已建立的 ID 至少有一個對應範圍、完成定義、人工驗收或附排除依據的非目標。簡單一對一需求直接把來源與驗收寫在同一項即可，不為格式完整強制建立 ID 或 N/A 矩陣。
- 「完成定義」優先列出限定取證範圍內已證實可用的 repo command、script 與可觀測通過結果；無法確認命令時寫清楚預期行為與執行 agent 應從哪些已知設定選擇驗證，不得發明命令或擴大搜尋。分析／文件型工作可用具體可核對的內容、來源與一致性判準驗收，不必為了形式硬造 grep／腳本。只有確實需要產品判斷且無法由既有自動化或可觀測結果裁定的項目才放「人工驗收」，並說明原因；不要硬塞無關 build 命令或人工 gate。
- 「待確認事項」逐項寫出未知資訊、影響與需要誰／何種證據決定；不得把推測寫成既定事實。沒有待確認事項時寫「無」，不要停下來反問使用者。
- 除非原需求指定其他語言，使用繁體中文。
