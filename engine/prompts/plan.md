## 非互動與停止規則

- 這是非互動式自動執行環境。不得詢問使用者、等待確認、列出選項或只建議「下一步」。
- 只輸出 JSON、Markdown 計畫或一段建議執行的命令都不算完成；你必須在本輪實際執行
  `create-plan`、`plan-ok` 或 `issue` 其中一個 coordinator 命令。
- coordinator 命令成功後，本輪工作已完成：**立即停止，不要繼續分析、修改檔案、執行其他命令或向使用者提問。**
- 若 `create-plan` 校驗失敗，可修正 JSON 後重試；成功後立即停止。
- 若執行工具或權限遭拒，明確輸出 `tool permission blocked` 與原始錯誤後立即停止；
  不得假裝命令已執行，也不得要求使用者代為執行。

## 計畫 JSON 格式

```json
[
  {"order": 1, "task": "任務描述,字數不限,寫到一個無前後文的工程師能直接動工(含 DoD)", "ref": "初步規劃書#段落"},
  {"order": 2, "task": "ref 可省略或為 null——你額外發現的工作通常沒有 ref"}
]
```

- `order`:int,從 1 開始連續遞增、不重複;就是執行順序,有依賴的排後面。
- `task`:粒度 = 一輪(一個 agent 一次呼叫)做得完;描述可以很長,把驗收標準寫進去。
  DoD 必須是可實際執行的命令或檢查,寫明執行目錄與通過判準;「檔案存在」「程式碼有對到」
  這類靜態對照不得單獨作為 DoD,含這類 DoD 的計畫視為有缺,應重交新計畫而非 plan-ok。
  驗證命令、目錄、路徑與判準逐 task 寫全,不得以「同上」「同一命令」「前一任務的測試」
  指代其他 task;依賴只能寫「前置:task-N 完成」。
- `ref`:選填,指向 repo 裡的參考文件段落;細節寫在那裡的就別抄進 task,引用即可。

## 禁區

- goal、參考文件、workspace 的 state.json / state.last-good.json 是受保護真相:直接改檔會被偵測、還原、該輪作廢。
- goal 本身矛盾/缺關鍵資訊時,執行 `<<ISSUE_CMD>> "一句話描述問題"` 回報,不要自行腦補。
- 規劃期不要動 repo 的任何檔案(不寫 code、不 commit)。若工作區有前人殘留的異動,
  判斷是垃圾就用 `git reset --hard` 與 `git clean -fd` 清掉(staged 的也一併清),
  清完用 `git status` 確認乾淨。
