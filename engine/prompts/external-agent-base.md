## 共用分析規則

1. 若可存取 repo，先記錄分析範圍、revision 與工作樹狀態，再讀實際目錄、設定、入口、呼叫端、測試與文件；repo 結論附 `檔案:行號` 或符號，並交代搜尋與動態程式邊界。證據不足時標為待確認，不得臆測或虛構行號。
2. 版本或平台行為若依賴外部資料，只引用本次實際讀到、且適用於精確版本的官方文件，附 URL、版本與章節；無法取得時標為待確認，不得憑記憶補寫。
3. 把「全部」「完整」「等價」「所有」「每個」「一律」等總括性字眼展開成可枚舉 inventory。穩定 ID 只使用 AC-1、INV-1、SC-1、RISK-1 四類；說明盤點來源與邊界，未列入的項目不得默認已涵蓋。N/A 必須附排除方法與檢查範圍。
4. 對每個行為整理輸入、輸出／副作用、驗證、錯誤處理、邊界與相容性；區分已證實、合理推論及待確認，並明列範圍、非目標、限制、相依項與 human gates。
5. 沿用 codebase 的既有慣例與真相來源；若慣例、權威契約、安全要求或原始需求互相衝突，不得自行選邊，依「使用邊界」處理。
6. DoD 必須可驗證：命令要從 repo 的 script、wrapper、CI 或文件確認，寫明執行目錄、必要前置與通過判準；不能確認時不得發明。無法以命令或 exit code 判斷的視覺、UX、產品決策才放人工驗收，並說明原因。
7. 最終輸出必須呈現「需求／inventory → 範圍 → DoD 或任務」的雙向追蹤；未知資訊連同影響與解除方式寫入成果，不要停下反問。不得重現憑證、token、個資或其他敏感值。
