套用方式：這是 Parallel Loop 的基礎 Plan 準備規則；它只補充人工並行審查所需證據，不改變下方 `plan.json` 的輸出 schema。

## Parallel 基礎 Plan 準備契約

- 你只產生不含 `stack` 的基礎 plan。不得自行推論、建議或輸出 `stack`；`stack` 必須由人類讀完任務邊界與 repo 證據後人工標註。
- 每個 task 除了既有目的、交付物與 DoD，還要明確寫出可從 repo 證實的 working set、會讀寫或產生的檔案／schema／生成物，以及與其他 order 的語意或資料依賴，讓人類能判斷是否可同批執行。
- 若驗證會使用 port、DB、cache、lock、外部服務、全域環境或其他共享資源，必須在各 task 內寫明隔離方式；找不到隔離證據時如實標示 human gate，不得猜測可以並行。
- 任務仍依依賴順序排列。不得宣稱任務已可安全並行，也不得為了製造並行機會而拆散不可獨立驗證的原子工作。
- 人工審查時只有 working set 不重疊、沒有生成物／語意依賴、驗證資源已隔離的連續 task 才能標成同一個 `stack`；任一條不確定就維持串行。
